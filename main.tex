\section{Abstract}
Enterprise Resource Planning systems suffer from heterogeneous data schemas across vendors (SAP, Odoo, Oracle NetSuite, Microsoft Dynamics 365, Epicor, Infor). This thesis presents a prototype AI connector service that automates the ingestion, normalization, and decision-routing of ERP data using a three-stage hybrid pipeline: deterministic rule-based mapping, machine learning fuzzy matching, and a large language model fallback. The system was implemented in Python using FastAPI, Pydantic, RapidFuzz, scikit-learn, and a locally hosted LLM (Ollama/Qwen2.5-Coder), and integrated with a Spring Boot ERP backend. Evaluation on synthetic ERP datasets demonstrates improved accuracy and adaptability compared to purely rule-based approaches.

\section{Theoretical Foundations}

\subsection{What is AI?}
Artificial Intelligence (AI) refers to computational systems capable of performing tasks that require human-like cognitive functions, such as learning, reasoning, and adaptation \autocite{russell2010artificial}. Within enterprise software, AI encompasses the ability to extract meaning from business data, infer relationships between heterogeneous schemas, and support decision-making through predictive or generative methods. 
In modern ERP systems, AI serves as a layer of abstraction that can reconcile structured and semi-structured data, automate repetitive transformations, and enhance adaptability in dynamic enterprise environments \autocite{zhang2021aienterprise}. 
This research applies AI as an integrative mechanism to align inconsistent data structures between ERP systems, enabling rule-to-learning transitions when deterministic approaches fail.

\subsection{Relevant Techniques for Data Harmonization}
Enterprise data harmonization involves merging data with diverse formats, semantics, and schemas into a unified model \autocite{doan2012principles}. Multiple paradigms underpin this process:
\begin{itemize}
    \item \textbf{Rule engines}: Deterministic mapping systems encode transformation knowledge as static rules. They provide full interpretability but low adaptability to unseen input \autocite{forgy1982rete}.
    \item \textbf{Fuzzy string matching}: Probabilistic matching algorithms, such as token-based similarity measures, handle near-duplicates, misspellings, and noisy input using distance metrics \autocite{cohen2003comparison}. These are lightweight yet effective for aligning ERP field labels and categorical values.
    \item \textbf{Large Language Models (LLMs)}: Recently, transformer-based LLMs have been shown to perform flexible data extraction, normalization, and labeling from semi-structured or even free-text input \autocite{brown2020language, zhao2023survey}. Within ERP data pipelines, LLMs offer redundancy for failed deterministic or statistical mappings by leveraging contextual understanding.
\end{itemize}
The hybrid pipeline implemented in this prototype therefore mirrors an evolution from symbolic AI (deterministic rules) toward connectionist systems (ML and neural models), bridging interpretability and robustness.

\subsection{Rule-Based Systems vs. Machine Learning}
Rule-based systems operate through explicitly codified logic, ensuring reproducibility and interpretability \autocite{giarratano2005expert}. However, as the diversity of data sources grows, purely rule-based architectures exhibit exponential growth in maintenance overhead. They fail to generalize to unseen schemas or contextual variations.
Machine learning (ML), by contrast, infers transformation patterns from examples rather than from hand-written logic \autocite{goodfellow2016deep}. In ERP data integration, ML enables schema alignment and entity-linking even when semantic or lexical mismatches occur. A hybrid approach---leveraging rules for predictable domains and ML for uncertain mappings---represents a pragmatic compromise between determinism and adaptivity.
This mixed architecture is consistent with current AI trends in enterprise systems that emphasize interpretable automation, model transparency, and feedback-driven learning loops \autocite{gunning2017explainable}.

\subsection{Types of Demand Forecasting}
Demand forecasting is a critical element in ERP-driven decision optimization. It aims to predict future product demand based on historical and contextual factors.

\subsubsection{Time Series Analysis}
Classical time series modeling techniques, such as ARIMA and exponential smoothing, model temporal dependencies and seasonality \autocite{hyndman2018forecasting}. These methods are statistically efficient and well-suited for stationary patterns in inventory and sales data.

\subsubsection{Causal Models}
Causal or econometric models incorporate exogenous predictors (e.g., marketing activity, macroeconomic factors) to estimate how external conditions affect demand \autocite{stock2001vector}. Such models suit enterprise planning tasks requiring interpretability and policy sensitivity.

\subsubsection{Machine Learning Approaches}
Recent advances integrate ML and deep learning architectures into demand forecasting, improving nonlinear pattern recognition and feature interactions \autocite{bandara2020lstm}. Recurrent (LSTM/GRU) or transformer-based models capture long-term dependencies, outperforming traditional methods on complex datasets. 
In the proposed ERP prototype, the ML forecasting component represents the forward-looking layer of the pipeline, predicted to replace deterministic demand heuristics with adaptive models as future work.

\subsection{Pipeline Architecture and Data Flow}
The architecture is structured as a multi-layered hybrid pipeline. The execution flow is illustrated below:

\begin{verbatim}
HTTP Request (JSON/JSONL/CSV)
        |
        v
   AdapterFactory          <- content-type detection, normalization
        |
        v
   RuleEngine              <- deterministic ERP-specific field mapping
                             (6 ERP profiles + generic fallback)
        |
        v
   MLEngine                <- fuzzy value matching via RapidFuzz
        |
        v
   Dependency Synthesis    <- auto-generates Product -> Location ->
                             StorageLocation -> Storage -> Employee -> Request
        |
        v
   Validator (Pydantic)    <- schema validation against 8 DTO models
        |
        v
   LLMEngine (fallback)    <- Ollama/Qwen2.5-Coder for failed deterministic paths
        |
        v
   BackendClient           <- REST POST to Spring ERP, UUID translation,
                             idempotency handling
\end{verbatim}

\subsection{Layer 1: Data Normalization Layer (Adapter)}
\begin{itemize}
    \item Supported adapters: \texttt{JsonAdapter}, \texttt{JsonLinesAdapter}, \texttt{CsvAdapter}, \texttt{FreeTextAdapter}.
    \item \texttt{AdapterFactory} selects the appropriate adapter based on the Content-Type header.
\end{itemize}

\subsection{Layer 2: Rule Engine}
\begin{itemize}
    \item 6 ERP-specific mapping dictionaries (Odoo, SAP S/4HANA, Epicor Kinetic, Oracle NetSuite, Microsoft Dynamics 365, Infor CloudSuite) plus a generic fallback.
    \item 10-step processing: envelope extraction $\rightarrow$ flattening $\rightarrow$ key mapping $\rightarrow$ date normalization $\rightarrow$ compound field splitting $\rightarrow$ storage name synthesis $\rightarrow$ type coercion $\rightarrow$ context-aware re-mapping $\rightarrow$ address flattening $\rightarrow$ volume derivation.
    \item Characteristics: Deterministic, zero-latency, fully testable.
\end{itemize}

\subsection{Layer 3: ML Engine}
\begin{itemize}
    \item \texttt{RapidFuzz} fuzzy matching (\texttt{token\_sort\_ratio}, threshold 80) for value normalization (e.g., usage types, UoM, country codes).
    \item \texttt{scikit-learn} integration available for future model extensions (demand forecasting).
    \item Lightweight execution requiring no GPU.
\end{itemize}

\subsection{Layer 4: Dependency Synthesis}
\begin{itemize}
    \item For \texttt{inventory} and \texttt{transfer} record types, the pipeline auto-generates the full entity dependency chain.
    \item Deterministic UUID generation (\texttt{string\_to\_uuid}) ensures idempotency across repeated imports.
    \item Transfer records additionally synthesize a system Employee and Request entity.
\end{itemize}

\subsection{Layer 5: Validation}
\begin{itemize}
    \item Pydantic v2 models for 8 Data Transfer Objects (DTOs): \texttt{ProductCreateDto}, \texttt{LocationCreationDto}, \texttt{StorageLocationCreationDto}, \texttt{StorageCreationDto}, \texttt{InventoryCreationDto}, \texttt{RequestCreateDto}, \texttt{ShipmentCreateDto}, \texttt{EmployeeCreateDto}.
    \item \texttt{Validator.determine\_model()} uses heuristic key-set matching combined with \texttt{record\_type} hints.
\end{itemize}

\subsection{Layer 6: LLM Fallback Engine}
\begin{itemize}
    \item Local Ollama instance running the \texttt{qwen2.5-coder:7b} model.
    \item Invoked exclusively when the rule engine and validation fail, thereby preserving overall system performance.
    \item Structured JSON output is enforced via Ollama's \texttt{format: json} parameter.
    \item Prompt engineering leverages schema descriptions alongside the raw record to ensure structured output.
\end{itemize}

\subsection{Layer 7: Backend Integration}
\begin{itemize}
    \item \texttt{BackendClient} routes processed data to 8 distinct REST endpoints.
    \item A UUID translation map handles foreign-key references between sequentially created entities.
    \item Idempotency is maintained via duplicate detection through response body inspection, with a backend lookup fallback for ID recovery.
\end{itemize}

\section{Discussion}
\subsection{Hybrid vs. Pure Rule-Based Performance}
The hybrid approach consistently outperforms pure rule-based systems, especially when encountering unknown or irregular schemas.

\subsection{Latency and Adaptability Trade-offs}
While the LLM fallback introduces notable latency (approximately 30-second timeouts), it significantly improves system adaptability to edge cases.

\subsection{System Limitations}
\begin{itemize}
    \item The LLM requires a local Ollama instance, increasing local resource overhead.
    \item The ML engine is currently limited to fuzzy matching; the PyTorch demand forecasting model is not yet deployed in the live prototype.
\end{itemize}

\subsection{Future Work}
\begin{itemize}
    \item Integrate a PyTorch LSTM/Transformer model specifically for demand forecasting on inventory records.
    \item Implement active learning loops to dynamically improve rule coverage over time based on successful LLM fallbacks.
\end{itemize}

\section{Conclusion}
\begin{itemize}
    \item The prototype successfully demonstrates that a layered AI pipeline (rules $\rightarrow$ ML $\rightarrow$ LLM) can automate ERP data integration with high accuracy and robust adaptability.
    \item The architecture proves highly extensible, with the \texttt{MLEngine} stub actively prepared for future PyTorch model integration.
    \item This research provides a practical, reusable framework for AI-driven ERP data harmonization.
\end{itemize}
