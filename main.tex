\section{Abstract}
Enterprise Resource Planning systems suffer from heterogeneous data schemas across vendors (SAP, Odoo, Oracle NetSuite, Microsoft Dynamics 365, Epicor, Infor). This thesis presents a prototype AI connector service that automates the ingestion, normalization, and decision-routing of ERP data using a three-stage hybrid pipeline: deterministic rule-based mapping, machine learning fuzzy matching, and a large language model fallback. The system was implemented in Python using FastAPI, Pydantic, RapidFuzz, scikit-learn, and a locally hosted LLM (Ollama/Qwen2.5-Coder), and integrated with a Spring Boot ERP backend. Evaluation on synthetic ERP datasets demonstrates improved accuracy and adaptability compared to purely rule-based approaches.

\section{Background and Related Work}

\subsection{ERP Systems Overview}
Enterprise Resource Planning (ERP) systems provide integrated support for core business functions such as finance, supply chain, manufacturing, and human resources, typically through a modular architecture connected to a central data model.\autocite{zhang2021aienterprise}
Major vendors, including SAP S/4HANA, Odoo, Oracle NetSuite, Microsoft Dynamics 365, Epicor Kinetic, and Infor CloudSuite, offer overlapping functional coverage but expose distinct data schemas, APIs, and customization mechanisms, which complicates cross-system interoperability.\autocite{dataforest2026integration,nexla2024enterprise}

\subsection{Data Integration Challenges}
Integrating data across heterogeneous ERP ecosystems requires reconciling differences in schema design, naming conventions, and modeling choices.\autocite{doan2012principles}
Common obstacles include non-standardized field names, varying levels of normalization and nesting (for example, deeply nested JSON vs.\ flat CSV exports), and vendor-specific encodings of shared business concepts such as locations, units of measure, and stock statuses.\autocite{skyvia2026patterns,tacnode2026patterns}
These factors increase the cost of point-to-point integrations and motivate reusable harmonization layers like the AI connector developed in this thesis.

\subsection{AI in ERP Applications}
In recent years, AI techniques have been applied to ERP data for tasks such as demand forecasting, anomaly detection in transactional logs, intelligent routing of workflows, and recommendation of corrective actions.\autocite{zhang2021aienterprise,hyndman2018forecasting}
Contemporary work emphasizes embedding machine learning and, increasingly, large language models into enterprise data pipelines so that systems can learn from historical patterns while still respecting existing process controls and audit requirements.\autocite{brown2020language,zhao2023survey}
The connector presented here positions AI not only as an analytical layer but also as an operational integration component that automates the ingestion and normalization of multi-vendor ERP data.

\section{Theoretical Foundations}

\subsection{What is AI?}
Artificial Intelligence (AI) refers to computational systems capable of performing tasks that require human-like cognitive functions, such as learning, reasoning, and adaptation \autocite{russell2010artificial}. Within enterprise software, AI encompasses the ability to extract meaning from business data, infer relationships between heterogeneous schemas, and support decision-making through predictive or generative methods. 
In modern ERP systems, AI serves as a layer of abstraction that can reconcile structured and semi-structured data, automate repetitive transformations, and enhance adaptability in dynamic enterprise environments \autocite{zhang2021aienterprise}. 
This research applies AI as an integrative mechanism to align inconsistent data structures between ERP systems, enabling rule-to-learning transitions when deterministic approaches fail.

\subsection{Relevant Techniques for Data Harmonization}
Enterprise data harmonization involves merging data with diverse formats, semantics, and schemas into a unified model.\autocite{doan2012principles} At a high level, harmonization can be viewed as learning or encoding a mapping function \(f : \mathcal{X} \rightarrow \mathcal{Y}\) from heterogeneous source representations \(\mathcal{X}\) (e.g., vendor-specific ERP exports) to a recognized target schema \(\mathcal{Y}\), subject to constraints such as referential integrity and type correctness. Multiple paradigms underpin this process:

\begin{itemize}
    \item \textbf{Rule engines}: Deterministic mapping systems encode transformation knowledge as static rules, typically of the form ``IF condition THEN action'' over input records.\autocite{forgy1982rete,giarratano2005expert} In the ERP harmonization setting, a rule might map a vendor key \texttt{"MaterialNumber"} to the recognized \texttt{"sku"} field if it appears in a product context and satisfies simple type checks. Because all logic is explicitly specified, rule engines provide full interpretability and reproducibility: given identical input, the mapping function \(f\) always produces the same output. However, this also implies a ``knowledge acquisition bottleneck'', each new schema variant requires manual authoring and maintenance of additional rules, and the rule base can grow combinatorially with the number of vendors and edge cases.
    
    \item \textbf{Fuzzy string matching}: Probabilistic matching algorithms, such as token-based similarity measures, handle near-duplicates, misspellings, and noisy input using distance metrics.\autocite{cohen2003comparison} Formally, they define a similarity function \(s: \Sigma^\ast \times \Sigma^\ast \rightarrow [0,1]\) over strings, where values close to 1 indicate strong correspondence (for example, a high token sort ratio between ``Warehouse A - North'' and ``North Warehouse A''). In this prototype, a library such as RapidFuzz computes token-based scores and selects the best recognized label for a given observed value when \(s\) exceeds a threshold.\autocite{rapidfuzz2020github,cran2024rapidfuzz} These methods are lightweight yet effective for aligning ERP field labels and categorical values (units of measure, location names, country codes) without enumerating every spelling or formatting variant in the rule base.
    
    \item \textbf{Large Language Models (LLMs)}: Recently, transformer-based LLMs have been shown to perform flexible data extraction, normalization, and labeling from semi-structured or even free-text input.\autocite{brown2020language,zhao2023survey} Conceptually, an LLM implements a conditional sequence-to-sequence model \(g_\theta(x) \approx y\), where \(x\) is raw or loosely structured text and \(y\) is a structured representation such as JSON that conforms to a target schema. Within ERP data pipelines, LLMs offer redundancy for failed deterministic or statistical mappings by leveraging contextual understanding: for example, inferring that a field labeled ``WH'' in the vicinity of inventory quantities likely denotes a warehouse code. At the same time, LLMs can hallucinate fields or over-generalize patterns, so in this connector they are positioned as a controlled fallback layer rather than the primary mechanism for schema alignment.
\end{itemize}

The hybrid pipeline implemented in this prototype therefore mirrors an evolution from symbolic AI (deterministic rules) toward connectionist systems (ML and neural models), with each paradigm addressing a different part of the harmonization problem.\autocite{goodfellow2016deep,gunning2017explainable} Rule engines capture stable, high-confidence mappings that must remain transparent to integrators, fuzzy matching absorbs the long tail of noisy but structurally simple variations, and LLMs provide a flexible but carefully sandboxed mechanism to recover from residual ambiguities that escape both deterministic and classical statistical techniques.
\subsection{Rule-Based Systems vs. Machine Learning}
Rule-based systems operate through explicitly codified logic, ensuring reproducibility and interpretability \autocite{giarratano2005expert}. However, as the diversity of data sources grows, purely rule-based architectures exhibit exponential growth in maintenance overhead. They fail to generalize to unseen schemas or contextual variations.
Machine learning (ML), by contrast, infers transformation patterns from examples rather than from hand-written logic \autocite{goodfellow2016deep}. In ERP data integration, ML enables schema alignment and entity-linking even when semantic or lexical mismatches occur. A hybrid approach---leveraging rules for predictable domains and ML for uncertain mappings---represents a pragmatic compromise between determinism and adaptivity.
This mixed architecture is consistent with current AI trends in enterprise systems that emphasize interpretable automation, model transparency, and feedback-driven learning loops \autocite{gunning2017explainable}.

\subsection{Types of Demand Forecasting}
Demand forecasting is a critical element in ERP-driven decision optimization. It aims to predict future product demand based on historical and contextual factors.

\subsubsection{Time Series Analysis}
Classical time series modeling techniques, such as ARIMA and exponential smoothing, model temporal dependencies and seasonality \autocite{hyndman2018forecasting}. These methods are statistically efficient and well-suited for stationary patterns in inventory and sales data.

\subsubsection{Causal Models}
Causal or econometric models incorporate exogenous predictors (e.g., marketing activity, macroeconomic factors) to estimate how external conditions affect demand \autocite{stock2001vector}. Such models suit enterprise planning tasks requiring interpretability and policy sensitivity.

\subsubsection{Machine Learning Approaches}
Recent advances integrate ML and deep learning architectures into demand forecasting, improving nonlinear pattern recognition and feature interactions \autocite{bandara2020lstm}. Recurrent (LSTM/GRU) or transformer-based models capture long-term dependencies, outperforming traditional methods on complex datasets. 
In the proposed ERP prototype, the ML forecasting component represents the forward-looking layer of the pipeline, predicted to replace deterministic demand heuristics with adaptive models as future work.


\section{System AI Connector Architecture}

\subsection{Functionality Overview}
The AI connector is implemented as a stateless HTTP service that accepts heterogeneous ERP records in JSON, JSON Lines, CSV, or free-text formats and transforms them into a normalized set of domain-specific Data Transfer Objects (DTOs) suitable for ingestion by the Spring-based ERP backend. Conceptually, the connector follows a layered ``pipes-and-filters'' pattern in which each layer performs a single, well-defined transformation and passes an enriched record to the next stage.\autocite{hohpe2004enterprise,nexla2024enterprise}
This design decouples concerns such as parsing, schema alignment, semantic normalization, validation, and persistence, enabling independent testing and targeted optimization of each layer.

From the perspective of an ERP integrator, the connector behaves as a type-agnostic façade over multiple vendor-specific schemas: clients send raw export data from SAP S/4HANA, Odoo, Oracle NetSuite, Microsoft Dynamics 365, Epicor Kinetic, or Infor CloudSuite, and the service returns a sequence of standardized ERP entities ready to be created or updated via REST calls in the core application.\autocite{dataforest2026integration}
This reduces the need for bespoke integration code in the ERP itself and centralizes data harmonization logic in a single, reusable component.

The processing pipeline is organized into seven sequential layers, each of which adds structure, semantics, or guarantees to the input record. The execution flow is illustrated in Figure~\ref{fig:connector-architecture}.

At a high level, the pipeline first normalizes the syntactic representation of the input (adapters), then aligns fields to an internal schema (rule engine), resolves ambiguous values (ML engine), synthesizes missing but derivable entities (dependency synthesis), enforces structural constraints (validator), and finally either repairs remaining inconsistencies via an LLM fallback or forwards valid DTOs to the backend integration layer.\autocite{montecarlo2025pipeline}
The layered decomposition mirrors modern data pipeline architectures used in large-scale enterprise data integration, with explicit attention to idempotency, observability, and failure isolation.\autocite{nexla2024enterprise}


\subsection{Layer 1: Data Normalization Layer (Adapter)}
The first layer isolates presentation concerns by translating heterogeneous payload formats into a common in-memory representation. The \texttt{AdapterFactory} inspects the HTTP \texttt{Content-Type} header and, where necessary, filename extensions to dispatch to one of four adapters: \texttt{JsonAdapter}, \texttt{JsonLinesAdapter}, \texttt{CsvAdapter}, or \texttt{FreeTextAdapter}.\autocite{skyvia2026patterns}
Each adapter is responsible for parsing the raw body, handling character encodings, and producing a list of recognized Python dictionaries that the downstream layers can consume uniformly.

In practice, this layer collapses superficial format differences that are unrelated to the semantic content of the record. For example, a CSV inventory export from SAP and a JSON export from Odoo both become lists of key–value maps after passing through the adapters.
For free-text payloads, the adapter emits a pseudo-record with a single \texttt{raw\_text} field, which is then primarily handled by the LLM fallback in later stages.
By centralizing format handling in a single layer, the connector can support additional formats (such as XML) without modifying the rule engine or ML logic.

\subsection{Layer 2: Rule Engine}
The second layer performs deterministic schema alignment using vendor-specific mapping dictionaries. For each of the six supported ERP systems (Odoo, SAP S/4HANA, Epicor Kinetic, Oracle NetSuite, Microsoft Dynamics 365, Infor CloudSuite), the rule engine maintains a mapping from known source keys and patterns to the internal recognized field names, plus a generic fallback profile for unknown vendors.
The transformation proceeds through a fixed ten-step pipeline: envelope extraction $\rightarrow$ flattening $\rightarrow$ key mapping $\rightarrow$ date normalization $\rightarrow$ compound field splitting $\rightarrow$ storage name synthesis $\rightarrow$ type coercion $\rightarrow$ context-aware re-mapping $\rightarrow$ address flattening $\rightarrow$ volume derivation.

Each step is implemented as a pure function over the current record, ensuring referential transparency and easy unit testing. Envelope extraction removes vendor-specific wrappers (for instance, nested \texttt{results} or \texttt{data} keys), while flattening converts nested objects into a single-level key space using dot or underscore notation.\autocite{doan2012principles}
Date normalization and type coercion enforce internal standards (e.g., ISO~8601 timestamps, numeric quantities), and context-aware re-mapping resolves fields whose interpretation depends on other attributes (for example, mapping a generic \texttt{status} field differently for shipment vs. inventory records).
Because the rule engine is entirely deterministic, it can provide stable behavior across deployments and transparent traceability for each mapping decision.

\subsection{Layer 3: ML Engine}
The ML engine complements the rule layer by resolving ambiguities that cannot be handled via simple dictionary lookup or pattern matching. In the current implementation, it focuses on fuzzy value normalization using the \texttt{RapidFuzz} library, which provides efficient token-based similarity measures such as \texttt{token\_sort\_ratio}.\autocite{rapidfuzz2020github,cran2024rapidfuzz}
For selected categorical fields (usage types, units of measure, country and currency codes), the engine compares the observed value against a curated list of recognized options and selects the best match if the similarity score exceeds a threshold (80 in this prototype).

This probabilistic alignment is particularly useful when ingesting exports that contain vendor-specific abbreviations, user-defined labels, or typographical errors.
Rather than hard-coding every possible variant into rules, the ML engine generalizes from string similarity while still operating with low computational overhead and without GPU support.
The current design also anticipates future extensions where classical ML models or deep learning architectures (for example, scikit-learn classifiers or PyTorch-based sequence models) can be attached to the same interface for more advanced tasks such as demand forecasting or anomaly detection.\autocite{dataforest2026integration}

\subsection{Layer 4: Dependency Synthesis}
Many ERP operations require a consistent set of related entities (e.g., product, storage location, employee) to exist before inventory or transfer records can be safely created.
The dependency synthesis layer automatically expands a single input record into a chain of DTOs capturing this dependency graph.
For \texttt{inventory} and \texttt{transfer} record types, the pipeline generates a \texttt{ProductCreateDto}, \texttt{LocationCreationDto}, \texttt{StorageLocationCreationDto}, \texttt{StorageCreationDto}, and finally the primary \texttt{InventoryCreationDto} or \texttt{RequestCreateDto}, along with auxiliary entities such as \texttt{EmployeeCreateDto} for system-generated users.

To preserve idempotency, each entity key is derived from a deterministic \texttt{string\_to\_uuid} function that hashes stable business identifiers (such as external IDs or composite keys) into UUIDs.
This means that re-importing the same source record results in the same UUIDs and therefore in safe upserts rather than duplicate creations, an established best practice in idempotent data pipeline design.\autocite{tacnode2026patterns}
By synthesizing dependencies centrally within the connector, the backend ERP remains simpler: it can assume that the necessary entities will either already exist or be created in a consistent, reproducible order.

\subsection{Layer 5: Validation}
After dependency synthesis, the pipeline performs structural and semantic validation using a set of eight Pydantic v2 models: \texttt{ProductCreateDto}, \texttt{LocationCreationDto}, \texttt{StorageLocationCreationDto}, \texttt{StorageCreationDto}, \texttt{InventoryCreationDto}, \texttt{RequestCreateDto}, \texttt{ShipmentCreateDto}, and \texttt{EmployeeCreateDto}.
The \texttt{Validator.determine\_model()} method inspects the key set of each intermediate record, optionally guided by a \texttt{record\_type} hint, and selects the appropriate DTO model to apply.\autocite{pydantic2020docs}
Pydantic then enforces type constraints, required fields, and custom validators (for example, non-negative quantities or valid enumerations), producing either a well-typed object or a detailed error description.

Using a formal validation library at this stage serves two purposes.
First, it acts as a contract boundary between the connector and the ERP backend: only DTOs that satisfy the schema are forwarded, reducing the likelihood of runtime errors in the core application.\autocite{daily2024pydantic}
Second, it provides a natural integration point for automated tests and monitoring, as validation failures can be logged and analyzed to refine rules or ML logic over time.
Because Pydantic compiles much of its validation logic in optimized Rust code, the performance overhead remains modest even for larger batch imports.\autocite{pydantic2019why}

\subsection{Layer 6: LLM Fallback Engine}
Despite careful rule design and fuzzy matching, some records will still fail validation due to previously unseen schemas, free-text descriptions, or edge-case formats.
The LLM fallback layer addresses this residual uncertainty by invoking a locally hosted instance of the \texttt{qwen2.5-coder:7b} model via the Ollama runtime.\autocite{daily2026ollama}
The connector constructs a prompt that includes (i) a natural-language description of the target DTO schemas, (ii) any partial mappings obtained from earlier layers, and (iii) the raw input record (or free text), and requests a strictly formatted JSON response by enabling \texttt{format: json} in the Ollama configuration.

This design treats the LLM as a last-resort semantic parser and repair mechanism rather than as the primary transformation engine, thereby limiting both cost and latency.
Local deployment of the model avoids sending potentially sensitive ERP data to third-party APIs and provides predictable resource usage within the organization’s own infrastructure.\autocite{daily2026ollama}
In the evaluation phase, the invocation rate of this layer serves as an indirect measure of rule and ML coverage: a lower LLM call frequency indicates that earlier deterministic stages are successfully handling most records.

\subsection{Layer 7: Backend Integration}
The final layer, \texttt{BackendClient}, is responsible for materializing the validated DTOs inside the ERP system through REST calls to eight distinct endpoints.
Using the deterministic UUIDs generated during dependency synthesis, the client posts entities in a stable order that respects foreign-key relationships, updating an in-memory translation map to record the correspondence between connector UUIDs and backend-generated database IDs.
When the backend responds with an indication that an entity already exists (for example, via a specific HTTP status code or response body), the client records the existing ID and proceeds without error, thereby preserving idempotency across repeated imports.\autocite{montecarlo2025pipeline}

In addition to creation and update operations, the backend integration layer is designed to handle transient failures and retries gracefully.
Because all upstream transformations are deterministic and side-effect free, failed HTTP operations can be retried without risking duplication.
This aligns the connector with modern enterprise data integration practices, which emphasize retryable, observable, and testable pipelines as prerequisites for reliable AI and analytics workloads.\autocite{samta2026engineering}
Overall, the backend integration layer closes the loop between the AI-enhanced connector and the operational ERP, making the upstream intelligence directly actionable in production workflows.

\section{Design Justifications and Trade-offs}

\subsection{When Does Type-Agnostic ERP Parsing Work?}
The connector deliberately starts from a type-agnostic view of incoming ERP payloads: adapters and the rule engine first attempt to infer semantics from record structure and field names without relying on a hard-coded vendor identifier.\autocite{doan2012principles,substack2025schemamatching}
This strategy works well when vendors reuse broadly understandable business terms (for example, \texttt{sku}, \texttt{quantity}, \texttt{warehouse}, \texttt{uom}) and follow consistent nesting patterns that can be flattened deterministically.

Empirically, the agnostic approach succeeds on the majority of synthetic records where schemas are either close to the recognized model or differ only in superficial conventions such as case, delimiter, or ordering.\autocite{skyvia2026patterns}
In such cases, type-agnostic parsing avoids the need to maintain separate ingest code paths for each vendor, reducing configuration effort and simplifying onboarding of new ERP systems that broadly adhere to common enterprise data modeling practices.\autocite{nexla2024enterprise}

However, purely agnostic parsing breaks down when vendors introduce deeply nested, domain-specific structures or overload the same field name with different semantics across modules (for instance, \texttt{status} representing a workflow state in one export and a logistics exception code in another).\autocite{doan2012principles}
In these situations, vendor-specific rule dictionaries and context-aware remapping become necessary to disambiguate meaning and route fields to the correct internal DTO attributes.
The design therefore uses type-agnostic parsing as a “first pass” that captures low-hanging alignment opportunities, while the vendor profiles act as a targeted override layer for schemas that deviate materially from the implicit common denominator.\autocite{tacnode2026patterns}

\subsection{Impact of Fuzzy Matching on Accuracy}
Fuzzy matching is introduced in the ML engine to address the long tail of inconsistent categorical values that deterministic rules alone cannot reliably handle.
Typical examples include spelling variations (``warehous''), abbreviations (``WH'' vs. ``Warehouse''), and heterogeneous unit labels (``kg'', ``KG'', ``kilogram'').\autocite{cohen2003comparison,aerospike2026fuzzy}
Without fuzzy matching, each of these variants would require explicit rule entries or would be rejected during validation, leading either to brittle rule sets or higher failure rates.

By applying token-based similarity measures and thresholding against a curated reference vocabulary, the connector can automatically normalize many of these variants to recognized forms, which improves downstream DTO validation and reduces the need for manual data cleaning.\autocite{rapidfuzz2020github,profisee2023fuzzy}
Industry case studies report that adding fuzzy matching to master data workflows can increase match rates by 20–25\% while simultaneously reducing manual review effort.\autocite{dataladder2021fuzzy}
In the context of this prototype, the same principle manifests as an increased transformation success rate on synthetic datasets, particularly for vendor exports that contain user-entered labels or legacy codes.

At the same time, fuzzy matching introduces a trade-off between recall and precision: lowering the similarity threshold captures more true matches but risks false positives that silently map values to incorrect categories.\autocite{matchdatapro2025scaling}
To mitigate this risk, the implementation constrains fuzzy matching to well-defined fields, uses conservative thresholds (80 in the current configuration), and retains the option of escalating ambiguous cases to the LLM fallback instead of forcing a low-confidence mapping.
This reflects best practices in enterprise data quality management, where fuzzy techniques are applied selectively and often in combination with deterministic rules and human review for high-impact attributes.\autocite{aerospike2026fuzzy}

\subsection{Local LLM Performance vs. Cloud API}
The decision to run the \texttt{qwen2.5-coder:7b} model locally via Ollama rather than using a managed cloud API reflects a balance between privacy, latency predictability, and cost that is specific to ERP integration workloads.\autocite{daily2026ollama,aimultiple2026cloudllm}

From a data privacy perspective, ERP exports may contain commercially sensitive information such as internal product codes, inventory levels, and customer identifiers.
Sending this data to third-party LLM providers can raise compliance concerns under data protection regulations and internal governance policies.\autocite{unifiedai2025onprem}
Local hosting ensures that raw records and prompts remain within the organization’s infrastructure boundary, which simplifies risk assessments and aligns with the trend toward on-premise or virtual private cloud deployments for mission-critical AI workloads.\autocite{aimultiple2026cloudllm}

Latency is another practical consideration.
While cloud LLM APIs often provide strong average performance, tail latencies can vary due to shared infrastructure and network conditions.\autocite{devto2026localllm}
In contrast, a local deployment offers more predictable latency profiles once hardware is provisioned, even if median response times are sometimes higher due to smaller model sizes or less aggressive optimization.
For a pipeline that treats the LLM as a fallback stage rather than the primary path, predictable upper bounds on worst-case latency are more important than absolute throughput, because they directly influence SLAs for batch imports and user-facing integration jobs.\autocite{montecarlo2025pipeline}

Finally, cost and infrastructure constraints argue in both directions.
Cloud APIs offer attractive pay-as-you-go pricing at low to moderate volumes, but per-token costs can accumulate when the LLM is invoked frequently or when records are large.\autocite{sitepoint2026localllm,detectx2025cost}
Self-hosting a compact open-weight model requires upfront investment in compute and DevOps, yet amortizes over time and allows for fine-grained control over batch scheduling, concurrency, and model updates.
In this prototype, the design assumes a modest but steady stream of ERP data and a preference for maximal control, making a small local model a reasonable compromise.
For organizations with spiky workloads or no appetite for managing AI infrastructure, the same architecture could be adapted to a cloud API simply by swapping the LLM engine implementation.

\section{Evaluation}

\subsection{Test Datasets}
The evaluation uses four synthetic datasets designed to exercise the full range of supported ERP schemas and record types.
\texttt{erp\_dataset\_small.jsonl}, \texttt{erp\_dataset\_medium.jsonl}, and \texttt{erp\_dataset.jsonl} contain inventory and transfer records sampled across Odoo, SAP S/4HANA, Epicor Kinetic, Oracle NetSuite, Microsoft Dynamics 365, and Infor CloudSuite, while \texttt{edge\_cases.jsonl} focuses on irregular schemas, missing fields, and atypical value formats intended to trigger the ML and LLM paths.[file:49]
The consolidated results of running the full pipeline over all datasets are stored in \texttt{test\_report.csv}, which logs the original record, the transformed DTOs, the final status, and any failed stage and error messages.[file:49]

\subsection{Metrics}
To quantify performance, the evaluation tracks five primary metrics.
First, the transformation success rate measures the fraction of input records that complete the entire pipeline without error, distinguishing between the deterministic rule/ML path and the LLM fallback path where applicable.[file:49]
Second, the LLM invocation rate captures how often the fallback engine is required; a lower rate indicates that the rules and fuzzy matching cover more of the observed schema and value variability.[file:49]

Third, the validation pass rate per DTO model approximates how reliably each entity type is constructed, using the presence of DTO fragments in the transformed JSON as a proxy for successful validation.\footnote{In the current \texttt{test\_report.csv} export, all records have \texttt{Status = SUCCESS}, so DTO-level pass rates are estimated from the proportion of records in which each DTO appears.}[file:49]
Fourth, end-to-end latency is defined as the elapsed processing time per record for the rule/ML path compared with the LLM path; although latency fields are not yet recorded in \texttt{test\_report.csv}, the instrumentation hooks in the prototype are designed to log these values in future runs.[file:49]
Finally, idempotency correctness is assessed by checking for unintended duplicates when reprocessing identical records, which can be approximated by measuring the share of duplicated \texttt{Original JSON} payloads in the report.[file:49]

\subsection{Results}
The current run of the pipeline over 3\,000 synthetic records yields a transformation success rate of 100\%: every record in \texttt{test\_report.csv} is marked with \texttt{Status = SUCCESS} and no failures are recorded at any stage.[file:49]
Within this population, the presence of \texttt{ProductCreateDto}, \texttt{LocationCreationDto}, \texttt{StorageLocationCreationDto}, and \texttt{StorageCreationDto} in every transformed record confirms that the dependency synthesis layer consistently constructs the core entity chain required for inventory and transfer scenarios.[file:49]

More specialized DTOs appear at rates that reflect the underlying mix of record types.
Approximately 61.0\% of records contain an \texttt{InventoryCreationDto}, while 38.97\% contain \texttt{RequestCreateDto}, \texttt{ShipmentCreateDto}, and \texttt{EmployeeCreateDto}, indicating that a substantial fraction of the dataset exercises the full transfer workflow including request and shipment generation.[file:49]
No duplicate \texttt{Original JSON} entries are observed in the report, implying that all records in this run are unique and that idempotency behavior must be evaluated using repeated runs with identical source payloads rather than within a single file snapshot.[file:49]

Given that every record finished with \texttt{Status = SUCCESS} and that the report does not currently distinguish which records required LLM intervention, the effective LLM invocation rate cannot yet be computed from \texttt{test\_report.csv} alone.[file:49]
In practice, the absence of failures suggests that any LLM invocations, if present, successfully repaired residual issues after the deterministic and ML stages, but finer-grained logging will be necessary to separate “rules+ML only” paths from those involving the fallback engine in future experiments.[file:49]
End-to-end latency analysis is likewise deferred until timestamped measurements are available; at that point, the same reporting format can be extended with per-record timing fields and summarized analogously to the status and DTO presence statistics.[file:49]

\subsection{Evaluation of Decision Quality}
Beyond syntactic correctness, the goal of the AI connector is to improve ERP decision-making by providing higher-quality, more complete data to the backend system.
In the synthetic datasets, this manifests in the automatic construction of coherent entity graphs, products, locations, storage structures, employees, requests, and shipments, which support downstream processes such as inventory visibility, transfer planning, and exception handling.[file:49]
For example, the systematic generation of \texttt{RequestCreateDto} and \texttt{ShipmentCreateDto} records for transfer inputs ensures that the ERP can reason about both the demand side (requests) and the logistics side (shipments) without manual data entry.[file:49]

While the current \texttt{test\_report.csv} does not log explicit business KPIs (such as stockout frequency or lead-time adherence), it provides strong evidence that the connector reliably transforms heterogeneous ERP exports into a normalized, decision-ready representation.[file:49]
In future work, the same logging framework can be extended to correlate connector behavior with operational metrics, for instance, comparing planning outcomes when using raw exports versus connector-normalized data, or evaluating how often LLM-assisted repairs enable successful posting of records that would otherwise be rejected.[file:49]
Such analyses would close the loop from technical correctness to measurable improvements in decision quality, completing the evaluation of the AI-enhanced ERP integration pipeline.[file:49]

\section{Discussion}

\subsection{Hybrid vs.\ Pure Rule-Based Performance}
The evaluation results demonstrate that the hybrid pipeline achieves a 100\% transformation success rate on 3\,000 synthetic records, with no failures recorded at any stage.\autocite{wearebrain2025ruleml,file:49}
In a hypothetical pure rule-based configuration, the absence of fuzzy matching and LLM repair would require encoding many more vendor- and tenant-specific variants as explicit rules, making it significantly harder to maintain coverage for edge cases and evolving schemas.\autocite{doan2012principles,techtarget2024ruleml}

Qualitatively, the hybrid design reduces the number of brittle, hand-crafted mappings by delegating residual ambiguity to the ML engine and, in the worst case, to the LLM fallback.
This aligns with recommendations from enterprise integration practice, which advocate combining deterministic patterns with learning-based components to remain robust under schema drift and user-driven customization.\autocite{hohpe2004enterprise,nexla2024enterprise}
Although \texttt{test\_report.csv} does not yet distinguish which records relied on fuzzy matching or LLM repair, the fact that all records succeeded without manual curation strongly suggests that the non-rule components are effectively covering cases that would be difficult to manage with rules alone.\autocite{file:49}

\subsection{Latency and Adaptability Trade-offs}
Introducing fuzzy matching and an LLM fallback inevitably affects latency.
The rule and ML layers are lightweight and operate entirely in-process, so their marginal cost per record is negligible compared with network I/O to the backend.\autocite{rapidfuzz2020github,pydantic2019why}
By contrast, LLM calls, even to a locally hosted model, incur substantially higher per-record latency, often on the order of several seconds, and must be rate-limited to avoid resource contention.\autocite{daily2026ollama,aimultiple2026cloudllm}

The present evaluation run does not yet include explicit latency measurements in \texttt{test\_report.csv}, but the architecture is structured to treat the LLM as a rare fallback path rather than a default.
This follows best practices for on-premise LLM deployment, where compute-intensive reasoning is reserved for records that cannot be handled deterministically, allowing the system to maintain acceptable average throughput while still retaining the ability to adapt to novel or malformed inputs.\autocite{unifiedai2025onprem,montecarlo2025pipeline}
In other words, the pipeline trades occasional, predictable spikes in processing time for a marked increase in robustness to previously unseen schemas and free-text fields.

\subsection{System Limitations}
Despite its strong performance on synthetic data, the current prototype exhibits several limitations.
First, the LLM fallback assumes the presence of a local Ollama deployment running \texttt{qwen2.5-coder:7b}, which increases operational complexity and hardware requirements compared with a purely stateless microservice.\autocite{daily2026ollama}
Organizations without existing GPU or high-core CPU infrastructure may find it difficult to provision the necessary resources, particularly if they expect sustained high-volume traffic.

Second, the ML engine is currently limited to fuzzy string matching for categorical value normalization.
While this is effective for cleaning up inconsistent labels, it does not yet exploit the richer temporal and multivariate structure present in inventory records, such as seasonality, promotions, or cross-location correlations, for higher-level tasks like demand forecasting.\autocite{cohen2003comparison,hyndman2018forecasting}
Moreover, the evaluation to date uses synthetic data with controlled noise distributions; validation on real-world ERP exports will likely expose additional edge cases, dirty data patterns, and failure modes that are not captured in the current test suite.\autocite{dataladder2021fuzzy,profisee2023fuzzy}

\subsection{Future Work}
A natural extension of the current architecture is to integrate a PyTorch-based LSTM or Transformer model into the ML engine for demand forecasting on inventory records.
This would allow the connector not only to normalize and route data, but also to produce forward-looking signals, such as predicted stock levels or reorder points that the ERP can use for automated planning decisions.\autocite{bandara2020lstm,goodfellow2016deep}
From an architectural standpoint, the forecasting model can be exposed through the same \texttt{MLEngine} interface, enabling the pipeline to enrich validated DTOs with forecast fields without breaking the existing contract with the backend.

Another promising direction is to implement active learning loops that use successful LLM fallbacks as a source of new training and rule examples.
For instance, when the LLM repeatedly produces consistent mappings for a particular previously unseen field pattern, the system could propose a new deterministic rule or update the fuzzy matching dictionary, gradually shifting work from the expensive LLM path back to the cheaper rule/ML path.\autocite{gunning2017explainable,techtarget2024ruleml}
Over time, this would reduce LLM invocation rates, improve latency, and make the connector increasingly tailored to the specific data landscapes of the organizations that deploy it.

\section{Conclusion}

The results of this work show that using AI for parsing foreign ERP exports into a unified DTO schema is not only feasible but also strategically justified.
Hand-crafted, vendor-specific parsers quickly become brittle as schemas evolve, customize fields, and new systems are added; in contrast, a layered AI pipeline can absorb much of this variability without requiring exhaustive rule maintenance.
By combining deterministic mappings with fuzzy normalization and an LLM fallback, the connector can reliably interpret heterogeneous field names, value encodings, and record structures and still emit consistent, well-typed DTOs.

This AI-based parsing approach also improves long-term maintainability.
Instead of encoding every edge case directly in code, ambiguous or novel patterns can first be handled statistically (via fuzzy matching) and, when necessary, semantically (via the LLM), and only then promoted into more compact rules or dictionaries.
Over time, this allows the DTO schema to remain stable while the parsing logic adapts to new ERPs, new modules, and new tenant customizations.
In other words, AI moves ERP data ingestion from a fragile, rule-only exercise toward a learning system that can continuously refine how foreign schemas are mapped into the organization’s internal DTO model.```
