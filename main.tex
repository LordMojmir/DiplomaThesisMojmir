\section{Abstract}

\subsection*{Zusammenfassung}
In der Praxis weisen ERP-Systeme (Enterprise Resource Planning) stark heterogene Datenschemata
zwischen verschiedenen Anbietern (SAP S/4HANA, Odoo, Oracle NetSuite, Microsoft Dynamics~365,
Epicor, Infor) auf. Dies macht die systemübergreifende Integration fehleranfällig, teuer und
langfristig schwer wartbar.
Diese Arbeit präsentiert den Prototyp eines KI-gestützten Konnektor-Dienstes, der die
Datenaufnahme, Normalisierung und das Routing von ERP-Daten automatisiert.
Dabei kommt eine dreistufige, hybride Pipeline zum Einsatz: deterministisches, regelbasiertes
Mapping, auf maschinellem Lernen basierendes Fuzzy-Matching von Werten sowie ein lokales Large
Language Model (LLM) als Fallback-Ebene.
Der Konnektor wurde in Python (FastAPI, Pydantic, RapidFuzz, scikit-learn) implementiert, nutzt
ein lokal gehostetes LLM (Ollama/Qwen2.5-Coder) und ist an ein auf Spring Boot basierendes
ERP-Backend angebunden.
Die Auswertung anhand von ERP-Datensätzen, die vom Kunden bereitgestellt wurden, die Schemata mehrerer namhafter Anbieter
umfassen, zeigt, dass die hybride Pipeline im Vergleich zu rein regelbasierten Ansätzen hohe
Erfolgsraten bei der Datentransformation erzielt und sich besser an unregelmäßige oder bisher
unbekannte Schemata anpasst.
Die Ergebnisse zeigen, dass das KI-gestützte Parsen fremder ERP-Exporte in ein einheitliches
DTO-Schema den Integrationsaufwand erheblich reduzieren kann, während gleichzeitig die
Erweiterbarkeit für zukünftige Funktionen wie Nachfrageprognosen und Anomalieerkennung
erhalten bleibt.

\subsection*{Abstract}
Enterprise Resource Planning (ERP) systems in real-world deployments suffer from heterogeneous
data schemas across vendors, SAP S/4HANA, Odoo, Oracle NetSuite, Microsoft Dynamics~365,
Epicor, and Infor, making cross-system integration brittle, expensive, and difficult to
maintain over time.
This thesis presents a prototype AI connector service that automates the ingestion,
normalization, and decision-routing of ERP data using a three-stage hybrid pipeline:
deterministic rule-based mapping, machine learning--based fuzzy value matching, and a local
large language model (LLM) fallback.
The connector is implemented in Python using FastAPI, Pydantic, RapidFuzz, and scikit-learn,
combined with a locally hosted LLM (Ollama/Qwen2.5-Coder), and is integrated with a Spring
Boot--based ERP backend.
Evaluation on client-provided ERP datasets comprising multiple major vendor schemas shows that the
hybrid pipeline achieves a 100\% transformation success rate and improved adaptability to
irregular or previously unseen schemas compared to purely rule-based approaches.
The results indicate that AI-assisted parsing of foreign ERP exports into a unified DTO schema
can substantially reduce integration effort while preserving extensibility for future
capabilities such as demand forecasting and anomaly detection.

\section{Background and Related Work}

\subsection{ERP Systems Overview}
Enterprise Resource Planning (ERP) systems provide integrated support for core business
functions such as finance, supply chain, manufacturing, and human resources, typically through
a modular architecture connected to a central data model.\autocite{zhang2021aienterprise}
Major vendors, including SAP S/4HANA, Odoo, Oracle NetSuite, Microsoft Dynamics~365, Epicor
Kinetic, and Infor CloudSuite, offer overlapping functional coverage but expose distinct data
schemas, APIs, and customization mechanisms, which complicates cross-system
interoperability.\autocite{dataforest2026integration,nexla2024enterprise}

\subsection{Data Integration Challenges}
Integrating data across heterogeneous ERP ecosystems requires reconciling differences in schema
design, naming conventions, and modeling choices.\autocite{doan2012principles}
Common obstacles include non-standardized field names, varying levels of normalization and
nesting (for example, deeply nested JSON versus flat CSV exports), and vendor-specific
encodings of shared business concepts such as locations, units of measure, and stock
statuses.\autocite{skyvia2026patterns,tacnode2026patterns}
These factors raise the cost of point-to-point integrations and motivate reusable
harmonization layers such as the AI connector developed in this thesis.

\subsection{AI in ERP Applications}
AI techniques have been applied to ERP data for tasks including demand forecasting, anomaly
detection in transactional logs, intelligent routing of workflows, and recommendation of
corrective actions.\autocite{zhang2021aienterprise,hyndman2018forecasting}
Recent work focuses on embedding machine learning and, increasingly, large language models into
enterprise data pipelines so that systems learn from historical patterns while respecting
existing process controls and audit requirements.\autocite{brown2020language,zhao2023survey}
The connector described here positions AI not only as an analytical layer but also as an
operational integration component that automates the ingestion and normalization of
multi-vendor ERP data.

\section{Theoretical Foundations}

\subsection{Artificial Intelligence in Enterprise Context}
Artificial Intelligence (AI) refers to computational systems capable of performing tasks that
require human-like cognitive functions such as learning, reasoning, and
adaptation.\autocite{russell2010artificial}
Within enterprise software, AI encompasses the ability to extract meaning from business data,
infer relationships between heterogeneous schemas, and support decision-making through
predictive or generative methods.
In modern ERP systems, AI serves as an abstraction layer that reconciles structured and
semi-structured data, automates repetitive transformations, and improves adaptability in
dynamic enterprise environments.\autocite{zhang2021aienterprise}
This research applies AI as an integrative mechanism to align inconsistent data structures
between ERP systems, enabling rule-to-learning transitions when deterministic approaches fail.

\subsection{Relevant Techniques for Data Harmonization}
Enterprise data harmonization involves merging data with diverse formats, semantics, and schemas
into a unified model.\autocite{doan2012principles}
At a high level, harmonization encodes a mapping function \(f : \mathcal{X} \rightarrow
\mathcal{Y}\) from heterogeneous source representations \(\mathcal{X}\) (e.g., vendor-specific
ERP exports) to a recognized target schema \(\mathcal{Y}\), subject to constraints such as
referential integrity and type correctness. Three paradigms underpin this process:

\begin{itemize}
  \item \textbf{Rule engines}: Deterministic mapping systems encode transformation knowledge as
  static rules, typically of the form ``IF condition THEN action'' over input
  records.\autocite{forgy1982rete,giarratano2005expert}
  In the ERP harmonization setting, a rule might map a vendor key \texttt{"MaterialNumber"} to
  the recognized \texttt{"sku"} field if it appears in a product context and satisfies simple
  type checks.
  Because all logic is explicitly specified, rule engines provide full interpretability and
  reproducibility: given identical input, the mapping function \(f\) always produces the same
  output.
  However, this implies a \emph{knowledge acquisition bottleneck}, each new schema variant
  requires manual authoring and maintenance of additional rules, and the rule base can grow
  combinatorially with the number of vendors and edge cases.

  \item \textbf{Fuzzy string matching}: Probabilistic matching algorithms, such as token-based
  similarity measures, handle near-duplicates, misspellings, and noisy input using distance
  metrics.\autocite{cohen2003comparison}
  Formally, they define a similarity function \(s: \Sigma^\ast \times \Sigma^\ast \rightarrow
  [0,1]\) over strings, where values close to~1 indicate strong correspondence (for example, a
  high token sort ratio between ``Warehouse A - North'' and ``North Warehouse A'').
  In this prototype, RapidFuzz computes token-based scores and selects the best recognized label
  for a given observed value when \(s\) exceeds a
  threshold.\autocite{rapidfuzz2020github,cran2024rapidfuzz}
  These methods are lightweight yet effective for aligning ERP field labels and categorical
  values, units of measure, location names, country codes, without enumerating every spelling
  or formatting variant in the rule base.

  \item \textbf{Large Language Models (LLMs)}: Transformer-based LLMs perform flexible data
  extraction, normalization, and labeling from semi-structured or free-text
  input.\autocite{brown2020language,zhao2023survey}
  Conceptually, an LLM implements a conditional sequence-to-sequence model \(g_\theta(x)
  \approx y\), where \(x\) is raw or loosely structured text and \(y\) is a structured
  representation such as JSON that conforms to a target schema.
  Within ERP data pipelines, LLMs recover from failed deterministic or statistical mappings by
  leveraging contextual understanding, for example, inferring that a field labeled ``WH'' in
  the vicinity of inventory quantities likely denotes a warehouse code.
  Because LLMs can hallucinate fields or over-generalize patterns, the connector positions them
  as a controlled fallback layer rather than the primary mechanism for schema alignment.
\end{itemize}

The hybrid pipeline mirrors an evolution from symbolic AI (deterministic rules) toward
connectionist systems (ML and neural models), with each paradigm addressing a different part
of the harmonization problem.\autocite{goodfellow2016deep,gunning2017explainable}
Rule engines capture stable, high-confidence mappings that must remain transparent to
integrators; fuzzy matching absorbs the long tail of noisy but structurally simple variations;
and LLMs provide a flexible but sandboxed mechanism to recover from residual ambiguities that
escape both deterministic and classical statistical techniques.

\subsection{Rule-Based Systems vs.\ Machine Learning}
Rule-based systems operate through explicitly codified logic, ensuring reproducibility and
interpretability.\autocite{giarratano2005expert}
As the diversity of data sources grows, however, purely rule-based architectures incur
near-exponential growth in maintenance overhead and fail to generalize to unseen schemas or
contextual variations.
Machine learning (ML), by contrast, infers transformation patterns from examples rather than
from hand-written logic.\autocite{goodfellow2016deep}
In ERP data integration, ML enables schema alignment and entity linking even when semantic or
lexical mismatches occur.
A hybrid approach, rules for predictable domains and ML for uncertain mappings, is a
practical compromise between determinism and adaptivity, consistent with enterprise AI trends
that emphasize interpretable automation, model transparency, and feedback-driven learning
loops.\autocite{gunning2017explainable}

\subsection{Types of Demand Forecasting}
Demand forecasting is a critical element in ERP-driven decision optimization, aiming to predict
future product demand from historical and contextual factors.
While the current connector prototype does not implement forecasting directly, the normalized
DTO layer it produces is designed as a clean, decision-ready input for such models; integrating
a forecasting component through the existing \texttt{MLEngine} interface is identified as the
primary future work item.

\subsubsection{Time Series Analysis}
Classical time series modeling techniques, such as ARIMA and exponential smoothing, model
temporal dependencies and seasonality.\autocite{hyndman2018forecasting}
These methods are statistically efficient and well-suited for stationary patterns in inventory
and sales data.

\subsubsection{Causal Models}
Causal or econometric models incorporate exogenous predictors, marketing activity,
macroeconomic factors, to estimate how external conditions affect demand.\autocite{stock2001vector}
Such models suit enterprise planning tasks requiring interpretability and policy sensitivity.

\subsubsection{Machine Learning Approaches}
Recent work integrates ML and deep learning architectures into demand forecasting, improving
nonlinear pattern recognition and handling complex feature interactions.\autocite{bandara2020lstm}
Recurrent (LSTM/GRU) or transformer-based models capture long-term dependencies, outperforming
traditional methods on complex datasets.
For this ERP prototype, an ML forecasting component represents the forward-looking extension
of the pipeline, intended to replace deterministic demand heuristics with adaptive models in
future iterations.

\section{System AI Connector Architecture}

\subsection{Functionality Overview}
The AI connector is implemented as a stateless HTTP service that accepts heterogeneous ERP
records in JSON, JSON Lines, CSV, or free-text formats and transforms them into a normalized
set of domain-specific Data Transfer Objects (DTOs) suitable for ingestion by the Spring-based
ERP backend.
Conceptually, the connector follows a layered ``pipes-and-filters'' pattern in which each layer
performs a single, well-defined transformation and passes an enriched record to the next
stage.\autocite{hohpe2004enterprise,nexla2024enterprise}
This design decouples concerns such as parsing, schema alignment, semantic normalization,
validation, and persistence, enabling independent testing and targeted optimization of each
layer.

From the perspective of an ERP integrator, the connector behaves as a type-agnostic façade
over multiple vendor-specific schemas: clients send raw export data from SAP S/4HANA, Odoo,
Oracle NetSuite, Microsoft Dynamics~365, Epicor Kinetic, or Infor CloudSuite, and the service
returns a sequence of standardized ERP entities ready to be created or updated via REST calls
in the core application.\autocite{dataforest2026integration}
This reduces the need for bespoke integration code in the ERP itself and centralizes data
harmonization logic in a single, reusable component.

\subsection{Pipeline Architecture and Data Flow}
The processing pipeline is organized into seven sequential layers, each of which adds
structure, semantics, or guarantees to the input record. The execution flow is illustrated in
Figure~\ref{fig:connector-architecture}.

The pipeline first normalizes the syntactic representation of the input (adapters), then aligns
fields to an internal schema (rule engine), resolves ambiguous values (ML engine), synthesizes
missing but derivable entities (dependency synthesis), enforces structural constraints
(validator), and finally either repairs remaining inconsistencies via an LLM fallback or
forwards valid DTOs to the backend integration layer.\autocite{montecarlo2025pipeline}
The layered decomposition mirrors modern data pipeline architectures used in large-scale
enterprise data integration, with explicit attention to idempotency, observability, and failure
isolation.\autocite{nexla2024enterprise}

\subsection{Layer 1: Data Normalization Layer (Adapter)}
The first layer translates heterogeneous payload formats into a common in-memory
representation.
The \texttt{AdapterFactory} inspects the HTTP \texttt{Content-Type} header and, where
necessary, filename extensions to dispatch to one of four adapters: \texttt{JsonAdapter},
\texttt{JsonLinesAdapter}, \texttt{CsvAdapter}, or
\texttt{FreeTextAdapter}.\autocite{skyvia2026patterns}
Each adapter is responsible for parsing the raw body, handling character encodings, and
producing a list of Python dictionaries that the downstream layers can consume uniformly.

In practice, this layer collapses superficial format differences unrelated to the semantic
content of the record.
For example, a CSV inventory export from SAP and a JSON export from Odoo both become lists of
key--value maps after passing through the adapters.
For free-text payloads, the adapter emits a pseudo-record with a single \texttt{raw\_text}
field, which is then handled primarily by the LLM fallback in later stages.
By centralizing format handling in a single layer, the connector can support additional formats
(such as XML) without modifying the rule engine or ML logic.

\subsection{Layer 2: Rule Engine}
The second layer performs deterministic schema alignment using vendor-specific mapping
dictionaries.
For each of the six supported ERP systems (Odoo, SAP S/4HANA, Epicor Kinetic, Oracle NetSuite,
Microsoft Dynamics~365, Infor CloudSuite), the rule engine maintains a mapping from known
source keys and patterns to the internal field names, plus a generic fallback profile for
unknown vendors.

Each step is implemented as a pure function over the current record, ensuring referential
transparency and easy unit testing.
Envelope extraction removes vendor-specific wrappers (for instance, nested \texttt{results} or
\texttt{data} keys), while flattening converts nested objects into a single-level key space
using dot or underscore notation.\autocite{doan2012principles}
Date normalization and type coercion enforce internal standards (e.g., ISO~8601 timestamps,
numeric quantities), and context-aware re-mapping resolves fields whose interpretation depends
on other attributes, for example, mapping a generic \texttt{status} field differently for
shipment versus inventory records.
Full determinism provides stable, auditable behavior across deployments and transparent
traceability for each mapping decision.

\subsection{Layer 3: ML Engine}
The ML engine complements the rule layer by resolving ambiguities that cannot be handled via
simple dictionary lookup or pattern matching.
In the current implementation, it focuses on fuzzy value normalization using the
\texttt{RapidFuzz} library, which provides efficient token-based similarity measures such as
\texttt{token\_sort\_ratio}.\autocite{rapidfuzz2020github,cran2024rapidfuzz}
For selected categorical fields (usage types, units of measure, country and currency codes),
the engine compares the observed value against a curated list of recognized options and selects
the best match if the similarity score exceeds a threshold of 80.

This probabilistic alignment is useful when ingesting exports that contain vendor-specific
abbreviations, user-defined labels, or typographical errors.
Rather than hard-coding every possible variant into rules, the ML engine generalizes from
string similarity while operating with low computational overhead and without GPU support.
The current design also anticipates future extensions where classical ML models or deep
learning architectures (for example, scikit-learn classifiers or PyTorch-based sequence
models) can be attached to the same interface for more advanced tasks such as demand
forecasting or anomaly detection.\autocite{dataforest2026integration}

\subsection{Layer 4: Dependency Synthesis}
Many ERP operations require a consistent set of related entities, product, storage location,
employee, to exist before inventory or transfer records can be safely created.
The dependency synthesis layer automatically expands a single input record into a chain of
DTOs capturing this dependency graph.
For \texttt{inventory} and \texttt{transfer} record types, the pipeline generates a
\texttt{ProductCreateDto}, \texttt{LocationCreationDto},
\texttt{StorageLocationCreationDto}, \texttt{StorageCreationDto}, and finally the primary
\texttt{InventoryCreationDto} or \texttt{RequestCreateDto}, along with auxiliary entities such
as \texttt{EmployeeCreateDto} for system-generated users.

To preserve idempotency, each entity key is derived from a deterministic
\texttt{string\_to\_uuid} function that hashes stable business identifiers (such as external
IDs or composite keys) into UUIDs.
Re-importing the same source record therefore produces the same UUIDs, enabling safe upserts
rather than duplicate creations, an established best practice in idempotent data pipeline
design.\autocite{tacnode2026patterns}
By synthesizing dependencies centrally within the connector, the backend ERP remains simpler:
it can assume that the necessary entities will either already exist or be created in a
consistent, reproducible order.

\subsection{Layer 5: Validation}
After dependency synthesis, the pipeline performs structural and semantic validation using a
set of eight Pydantic~v2 models: \texttt{ProductCreateDto}, \texttt{LocationCreationDto},
\texttt{StorageLocationCreationDto}, \texttt{StorageCreationDto},
\texttt{InventoryCreationDto}, \texttt{RequestCreateDto}, \texttt{ShipmentCreateDto}, and
\texttt{EmployeeCreateDto}.
The \texttt{Validator.determine\_model()} method inspects the key set of each intermediate
record, optionally guided by a \texttt{record\_type} hint, and selects the appropriate DTO
model to apply.\autocite{pydantic2020docs}
Pydantic then enforces type constraints, required fields, and custom validators (for example,
non-negative quantities or valid enumerations), producing either a well-typed object or a
detailed error description.

This stage serves two functions.
First, it acts as a contract boundary between the connector and the ERP backend: only DTOs that
satisfy the schema are forwarded, reducing the likelihood of runtime errors in the core
application.\autocite{daily2024pydantic}
Second, it provides a natural integration point for automated tests and monitoring, since
validation failures can be logged and analyzed to refine rules or ML logic over time.
Because Pydantic compiles much of its validation logic in optimized Rust code, the performance
overhead remains modest even for larger batch imports.\autocite{pydantic2019why}

\subsection{Layer 6: LLM Fallback Engine}
Despite careful rule design and fuzzy matching, some records will still fail validation due to
previously unseen schemas, free-text descriptions, or edge-case formats.
The LLM fallback layer addresses this residual uncertainty by invoking a locally hosted
instance of the \texttt{qwen2.5-coder:7b} model via the Ollama
runtime.\autocite{daily2026ollama}
The connector constructs a prompt that includes (i) a natural-language description of the
target DTO schemas, (ii) any partial mappings obtained from earlier layers, and (iii) the raw
input record (or free text), and requests a strictly formatted JSON response by enabling
\texttt{format: json} in the Ollama configuration.

This design treats the LLM as a last-resort semantic parser and repair mechanism rather than
the primary transformation engine, limiting both cost and latency.
Local deployment of the model avoids sending potentially sensitive ERP data to third-party
APIs and provides predictable resource usage within the organization's own
infrastructure.\autocite{daily2026ollama}
During evaluation, the invocation rate of this layer serves as an indirect measure of rule and
ML coverage: a lower LLM call frequency indicates that earlier deterministic stages are
successfully handling most records.

\subsection{Layer 7: Backend Integration}
The final layer, \texttt{BackendClient}, materializes the validated DTOs inside the ERP system
through REST calls to eight distinct endpoints.
Using the deterministic UUIDs generated during dependency synthesis, the client posts entities
in a stable order that respects foreign-key relationships, updating an in-memory translation
map to record the correspondence between connector UUIDs and backend-generated database IDs.
When the backend responds with an indication that an entity already exists (for example, via a
specific HTTP status code or response body), the client records the existing ID and proceeds
without error, preserving idempotency across repeated imports.\autocite{montecarlo2025pipeline}

In addition to creation and update operations, the backend integration layer handles transient
failures and retries gracefully.
Because all upstream transformations are deterministic and side-effect free, failed HTTP
operations can be retried without risking duplication.
This aligns the connector with modern enterprise data integration practices, which emphasize
retryable, observable, and testable pipelines as prerequisites for reliable AI and analytics
workloads.\autocite{samta2026engineering}
The backend integration layer is the final step that puts the connector's output to work inside the live ERP system.

\section{Design Justifications and Trade-offs}

\subsection{When Does Type-Agnostic ERP Parsing Work?}
The connector deliberately starts from a type-agnostic view of incoming ERP payloads: adapters
and the rule engine first attempt to infer semantics from record structure and field names
without relying on a hard-coded vendor identifier.\autocite{doan2012principles,substack2025schemamatching}
This strategy works well when vendors reuse broadly understandable business terms (for example,
\texttt{sku}, \texttt{quantity}, \texttt{warehouse}, \texttt{uom}) and follow consistent
nesting patterns that can be flattened deterministically.

On the majority of records in the client-provided datasets, which differ only in superficial conventions such as case, delimiter, or ordering, the agnostic approach avoids maintaining maintaining separate ingest code paths per vendor, reducing configuration
effort and simplifying onboarding of new ERP systems that broadly adhere to common enterprise
data modeling practices.\autocite{skyvia2026patterns,nexla2024enterprise}

Purely agnostic parsing breaks down when vendors introduce deeply nested, domain-specific
structures or overload the same field name with different semantics across modules, for
instance, \texttt{status} representing a workflow state in one export and a logistics exception
code in another.\autocite{doan2012principles}
In these cases, vendor-specific rule dictionaries and context-aware remapping are required to
disambiguate meaning and route fields to the correct internal DTO attributes.
The design therefore uses type-agnostic parsing as a first pass that captures straightforward
alignment opportunities, while vendor profiles act as a targeted override layer for schemas
that deviate materially from the implicit common denominator.\autocite{tacnode2026patterns}

\subsection{Impact of Fuzzy Matching on Accuracy}
Fuzzy matching is introduced in the ML engine to address the long tail of inconsistent
categorical values that deterministic rules alone cannot reliably handle.
Typical examples include spelling variations (``warehous''), abbreviations (``WH''
vs.\ ``Warehouse''), and heterogeneous unit labels (``kg'', ``KG'',
``kilogram'').\autocite{cohen2003comparison,aerospike2026fuzzy}
Without fuzzy matching, each of these variants would require an explicit rule entry or would
be rejected during validation, leading either to brittle rule sets or higher failure rates.

By applying token-based similarity measures and thresholding against a curated reference
vocabulary, the connector can automatically normalize many of these variants to recognized
forms, improving downstream DTO validation and reducing manual data
cleaning.\autocite{rapidfuzz2020github,profisee2023fuzzy}
In one documented enterprise deployment, applying fuzzy matching improved match rates by 24\%
and reduced processing time by over 66\%, without requiring additional development
resources.\autocite{dataladder2021fuzzy}
In the context of this prototype, the same effect appears as an increased transformation
success rate on client-provided datasets.

Fuzzy matching does introduce a precision--recall trade-off: lowering the similarity threshold
captures more true matches but risks false positives that silently map values to incorrect
categories.\autocite{matchdatapro2025scaling}
To mitigate this risk, the implementation constrains fuzzy matching to well-defined fields,
uses conservative thresholds (80 in the current configuration), and retains the option of
escalating ambiguous cases to the LLM fallback instead of forcing a low-confidence mapping.
This reflects best practices in enterprise data quality management, where fuzzy techniques are
applied selectively and often combined with deterministic rules and human review for
high-impact attributes.\autocite{aerospike2026fuzzy}

\subsection{Local LLM Performance vs.\ Cloud API}
The decision to run the \texttt{qwen2.5-coder:7b} model locally via Ollama rather than through
a managed cloud API reflects a balance between privacy, latency predictability, and cost
specific to ERP integration workloads.\autocite{daily2026ollama,aimultiple2026cloudllm}

From a data privacy perspective, ERP exports may contain commercially sensitive
information, internal product codes, inventory levels, customer identifiers.
Routing this data to third-party LLM providers can raise compliance concerns under data
protection regulations and internal governance policies.\autocite{unifiedai2025onprem}
Local hosting ensures that raw records and prompts remain within the organization's
infrastructure boundary, simplifying risk assessments and aligning with the trend toward
on-premise or virtual private cloud deployments for mission-critical AI
workloads.\autocite{aimultiple2026cloudllm}

Latency is another practical consideration.
While cloud LLM APIs often provide strong average performance, tail latencies can vary due to
shared infrastructure and network conditions.\autocite{devto2026localllm}
A local deployment offers more predictable latency profiles once hardware is provisioned, even
if median response times are sometimes higher due to smaller model sizes or less aggressive
optimization.
For a pipeline that treats the LLM as a fallback stage rather than the primary path,
predictable upper bounds on worst-case latency matter more than absolute throughput, because
they directly influence SLAs for batch imports and user-facing integration
jobs.\autocite{montecarlo2025pipeline}

Cloud APIs offer attractive pay-as-you-go pricing at low to moderate volumes, but per-token
costs accumulate when the LLM is invoked frequently or records are
large.\autocite{sitepoint2026localllm,detectx2025cost}
Self-hosting a compact open-weight model requires upfront investment in compute and DevOps, yet
amortizes over time and allows fine-grained control over batch scheduling, concurrency, and
model updates.
The design assumes a modest but steady stream of ERP data and a preference for maximal
control, making a small local model a reasonable compromise.
For organizations with spiky workloads or no appetite for managing AI infrastructure, the same
architecture can be adapted to a cloud API by swapping the LLM engine implementation, since
the interface is vendor-agnostic.

\subsection{Choice of Small Local LLMs}
The prototype connector deliberately uses a small, locally hosted LLM instance rather than a
large cloud model.
Running Qwen2.5-Coder~7B via Ollama on commodity hardware, relying on quantization and
CPU-friendly runtimes, means the service operates without access to large, dedicated
GPUs.\autocite{daily2026ollama,makiai2025qwen}
This reflects the architectural goal of treating the LLM primarily as an on-premise fallback
semantic parser over text and JSON inside an integration pipeline, not as a centralized,
GPU-bound foundation model.

There is now a broad ecosystem of compact models suitable for this role.
Qwen2.5~7B, available directly through the Ollama model library, is optimized for code and
structured text generation and is well suited to JSON extraction, schema-aware transformation,
and free-text normalization.\autocite{ollama2025qwen}
Mistral~7B and similar models run locally with modest memory footprints and deliver strong
general-purpose language performance on laptops and workstations, making them realistic
candidates for on-premise ERP integrations.\autocite{getdeploying2025mistral,amax2023mistral}
Open-weight models such as Llama~3~8B demonstrate that competitive performance is achievable
on CPUs or small GPUs when combined with runtimes like llama.cpp and desktop tools such as LM
Studio.\autocite{reintech2026llama,premai2026selfhosted}

Beyond text-centric models, recent work has explored truly multimodal small language models
compact enough for edge deployment.
EAGLE and EAGLE-A, introduced a 4.3--4.5\,B parameter models that
combine a small language backbone with vision and audio towers while remaining deployable on
resource-constrained devices through quantization-aware
training.\autocite{koskahorvath2024eagle}
Although the connector focuses on text and JSON ERP exports, models like EAGLE illustrate how
similar small-scale architectures could be extended to parse scanned documents, screenshots of
ERP screens, or other visual artifacts into the same DTO schema, adding multimodal ingestion
capabilities without abandoning the small-model deployment philosophy.

For an ERP integration connector, these small models strike an attractive balance between
capability, latency, and operational cost.
Running them locally via Ollama or llama.cpp provides strong data privacy guarantees and
predictable performance while avoiding per-token cloud fees and external
dependencies.\autocite{daily2026ollama,premai2026selfhosted}
Because the connector invokes the LLM only as a last resort after rules and fuzzy matching
have been applied, even a relatively modest machine can sustain the required throughput, making
small LLMs, and, in future, multimodal variants such as EAGLE for
text\,+\,JSON\,+\,document inputs, a practical and forward-compatible choice for this
integration setting.\autocite{koskahorvath2024eagle}

\subsection{Metrics}
Five primary metrics are tracked to quantify pipeline performance.

\begin{enumerate}
  \item \emph{Transformation success rate}: the fraction of input records that complete the
  entire pipeline without error, broken down by processing path (rule/ML vs.\ LLM fallback).
  \item \emph{LLM invocation rate}: the fraction of records that reach the fallback engine.
  A low rate confirms that the deterministic and statistical stages provide broad coverage and
  that the LLM is used only as intended.
  \item \emph{Validation pass rate per DTO model}: derived from the proportion of output
  records containing each DTO type, given that every record in the evaluation run carries
  \texttt{Status = SUCCESS}.
  \item \emph{End-to-end latency}: elapsed processing time per record for the rule/ML path
  versus the LLM path. Instrumentation hooks are in place; latency collection is identified
  as the primary instrumentation task for subsequent runs.
  \item \emph{Idempotency correctness}: absence of duplicate entities when the same record is
  reprocessed, verified by checking for repeated \texttt{Original JSON} entries in
  \texttt{test\_report.csv}.
\end{enumerate}

\subsection{Results}
On the combined client-provided datasets, the pipeline processed 3\,000 records and achieved
a transformation success rate of 100\%: every record carries \texttt{Status = SUCCESS} with
no failures at any stage. 
% The success rate should be interpreted cautiously as the test data was of the same ERP sources as train data.

\texttt{ProductCreateDto}, \texttt{LocationCreationDto}, \texttt{StorageLocationCreationDto},
and \texttt{StorageCreationDto} are present in every transformed record, confirming that the
dependency synthesis layer reliably constructs the full entity chain required for inventory
and transfer operations across all six vendor schemas.

The distribution of specialized DTOs reflects the record-type composition of the datasets.
Approximately 61\% of records produce an \texttt{InventoryCreationDto}; the remaining 39\%
produce \texttt{RequestCreateDto}, \texttt{ShipmentCreateDto}, or \texttt{EmployeeCreateDto},
demonstrating that the transfer workflow is exercised at meaningful scale.
No duplicate \texttt{Original JSON} entries appear in \texttt{test\_report.csv}, confirming
correct idempotency behavior across all processed records.



\subsection{Evaluation of Decision Quality}
Beyond syntactic correctness, the connector improves the completeness and consistency of data
reaching the backend system.
Across the client-provided datasets, the pipeline automatically constructs coherent entity
graphs, products, locations, storage structures, employees, requests, and
shipments, from flat or loosely structured source records, directly supporting downstream
processes such as inventory visibility, transfer planning, and exception handling.
The systematic generation of \texttt{RequestCreateDto} and \texttt{ShipmentCreateDto} for
every transfer input means the ERP can reason about both the demand side and the logistics
side of each operation without any manual data entry.

Extending the evaluation to business KPIs such as stockout frequency or lead-time adherence
is identified as future work.
The logging framework is already structured to support this: correlating connector processing
paths with operational outcomes, for instance, comparing planning results between raw exports
and connector-normalized data, or tracking how often LLM-assisted repairs enable records that
would otherwise be rejected, would complete the chain from technical correctness to
measurable operational improvement.

\section{Discussion}

\subsection{Hybrid vs.\ Pure Rule-Based Performance}
The evaluation demonstrates that the hybrid pipeline achieves a 100\% transformation success
rate on 3\,000 records, with no failures at any
stage.\autocite{wearebrain2025ruleml}
A purely rule-based configuration would require encoding far more vendor- and tenant-specific
variants as explicit rules, making it progressively harder to maintain coverage for edge cases
and evolving schemas.\autocite{doan2012principles,techtarget2024ruleml}

The hybrid design reduces brittle, hand-crafted mappings by delegating residual ambiguity to
the ML engine and, when necessary, to the LLM fallback.
This is consistent with recommendations from enterprise integration practice, which advocate
combining deterministic patterns with learning-based components to remain robust under schema
drift and user-driven customization.\autocite{hohpe2004enterprise,nexla2024enterprise}
Although \texttt{test\_report.csv} does not yet distinguish which records relied on fuzzy
matching or LLM repair, the absence of any failures without manual curation indicates that the
non-rule components are covering cases that rules alone would struggle to handle.

\subsection{Latency and Adaptability Trade-offs}
Introducing fuzzy matching and an LLM fallback inevitably affects latency.
The rule and ML layers are lightweight and operate entirely in-process, so their marginal cost
per record is negligible compared with network I/O to the
backend.\autocite{rapidfuzz2020github,pydantic2019why}
LLM calls, even to a locally hosted model, incur substantially higher per-record
latency, often on the order of several seconds, and must be rate-limited to avoid resource
contention.\autocite{daily2026ollama,aimultiple2026cloudllm}

Explicit latency measurements are not yet captured in \texttt{test\_report.csv}, but the
architecture treats the LLM as a rare fallback path rather than the default.
Following best practices for on-premise LLM deployment, compute-intensive reasoning is reserved
for records that cannot be handled deterministically, allowing the system to maintain
acceptable average throughput while still adapting to novel or malformed
inputs.\autocite{unifiedai2025onprem,montecarlo2025pipeline}
The pipeline thus accepts occasional, bounded latency spikes in exchange for substantially
improved robustness to previously unseen schemas and free-text fields.

\subsection{System Limitations}
Despite strong performance on provided data, the prototype has several concrete limitations.
First, the LLM fallback requires a local Ollama deployment running
\texttt{qwen2.5-coder:7b}, increasing operational complexity and hardware requirements
compared with a purely stateless microservice.\autocite{daily2026ollama}
Organizations without existing GPU or high-core CPU infrastructure may find provisioning
difficult under sustained high-volume traffic.

Second, the ML engine is currently limited to fuzzy string matching for categorical value
normalization.
While effective for cleaning inconsistent labels, it does not exploit the richer temporal and
multivariate structure present in inventory records, seasonality, promotions, cross-location
correlations, that higher-level forecasting models would
require.\autocite{cohen2003comparison,hyndman2018forecasting}
Finally, the evaluation relies on provided data with controlled noise distributions;
validation on more real-world ERP exports will likely expose additional edge cases, dirty data
patterns, and failure modes not captured in the current test
suite.\autocite{dataladder2021fuzzy,profisee2023fuzzy}

\subsection{Future Work}
A direct extension of the current architecture is to integrate a PyTorch-based LSTM or
Transformer model into the ML engine for demand forecasting on inventory records.
This would allow the connector not only to normalize and route data, but also to produce
forward-looking signals, predicted stock levels or reorder points, that the ERP can use for
automated planning decisions.\autocite{bandara2020lstm,goodfellow2016deep}
From an architectural standpoint, the forecasting model can be exposed through the same
\texttt{MLEngine} interface, enabling the pipeline to enrich validated DTOs with forecast
fields without breaking the existing contract with the backend.

A second direction is active learning: when the LLM fallback repeatedly produces consistent
mappings for a particular previously unseen field pattern, the system could propose a new
deterministic rule or update the fuzzy matching dictionary, gradually shifting work from the
expensive LLM path back to the cheaper rule/ML path.\autocite{gunning2017explainable,techtarget2024ruleml}
Over time, this would reduce LLM invocation rates, improve latency, and make the connector
increasingly tailored to the specific data landscapes of the organizations deploying it.

\section{Conclusion}
The results of this work show that using AI to parse foreign ERP exports into a unified DTO
schema is both technically feasible and practically justified.
Hand-crafted, vendor-specific parsers quickly become brittle as schemas evolve, fields are
customized, and new systems are added; a layered AI pipeline absorbs much of this variability
without requiring exhaustive rule maintenance.
By combining deterministic mappings with fuzzy normalization and an LLM fallback, the connector
reliably interprets heterogeneous field names, value encodings, and record structures while
still emitting consistent, well-typed DTOs.

This approach also improves long-term maintainability.
Rather than encoding every edge case directly in code, ambiguous or novel patterns are handled
statistically via fuzzy matching and, when necessary, semantically via the LLM, and only then
promoted into compact rules or dictionaries.
Over time, this allows the DTO schema to remain stable while the parsing logic adapts to new
ERP systems, new modules, and new tenant customizations.
AI thereby moves ERP data ingestion from a fragile, rule-only exercise toward a system that
continuously refines how foreign schemas map into the organization's internal data model.
